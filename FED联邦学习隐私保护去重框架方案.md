# 联邦LSH桶划分+候选集密文交换 详细解析（中心化版本）

该方案是为**跨机构中心化联邦协同去重**设计的隐私保护方案，核心是引入联邦主节点，将FED的全局LSH去重拆解为**各机构本地全流程FED计算**+**各机构向主节点提交密文候选集**+**主节点完成全局相似度匹配与结果聚合**，既保留FED原版的GPU加速、可复用哈希、矩阵化相似度对比等核心优势，又通过**最小信息传输+密文保护+主节点安全防护**实现隐私保障，且去重精度与原版FED完全一致（Jaccard≥0.96），是兼顾**隐私性、效率、工程落地性**的方案，适配企业/高校间的大模型语料协同训练场景。

该方案的核心特点是**中心化**（新增联邦主节点，负责全局统筹、数据汇总与计算、结果分发）、**最小数据传输**（各机构仅向主节点传输密文签名+无语义ID，无原始语料/完整签名矩阵）、**主节点安全可控**（通过多重防护机制规避单点风险），对FED原版的改造集中在**跨机构数据交互层**和**主节点计算层**，底层GPU计算逻辑完全复用，性能损耗仅10%~15%（主要来自GPU并行加解密和机构与主节点的网络传输）。

## 一、方案核心设计前提

1. **主节点与各机构算力对齐**：联邦主节点部署与FED兼容的高性能GPU集群（支持CUDA核函数、GPU并行计算、FED原版的LSH桶划分规则），各参与协同的机构也需部署兼容FED的GPU集群，确保本地特征生成与主节点全局计算的兼容性；

2. **全局参数统一**：由主节点统一制定并下发FED的核心超参数（n-gram长度、哈希函数参数p/q、LSH带数b/每行带数r、相似度阈值θ、桶数K=k√N），所有机构严格遵循该参数，确保各机构本地生成的签名矩阵、桶ID具有**全局一致性**，为主节点的全局匹配奠定基础；

3. **隐私基础保障**：各机构仅持有自身语料，无任何原始语料/明文特征的跨机构直接传输，所有与主节点的交互数据均为**轻量密文+无语义标识**，主节点仅负责密文处理与计算，不存储原始语料。

## 二、方案整体执行流程

该流程分为**本地预处理**、**主节点协同计算**、**结果分发与本地去重**三个阶段，共6个核心步骤，各机构负责本地GPU密集型计算，主节点负责全局统筹与跨机构相似度匹配，流程如下：

### 阶段1：本地全流程FED去重（无跨机构数据传输，各机构独立完成）

此阶段完全复用FED原版的GPU加速逻辑，核心目标是**完成本地语料去重**+**生成全局兼容的签名矩阵/桶ID**，为主节点协同计算做准备。

1. **步骤1：本地语料预处理**

各机构按主节点下发的统一参数，对自有语料做预处理（NFC归一化、过滤短文档<200字符、JSONL格式解析），并通过CPU并行加载语料到双缓冲区（文本缓冲区+文档索引缓冲区），缓冲区大小适配本地GPU显存。

1. **步骤2：本地GPU加速特征生成**

各机构将缓冲区数据传输到本地GPU集群，按FED原版完成**shingling（5-gram分片）**+**可复用哈希函数计算**+**MinHash签名矩阵生成**+**LSH桶划分**：

- 用FED的部分可复用非加密哈希函数生成shingle哈希值，签名生成速度较CPU基线快260倍；

- 按主节点统一的全局参数将签名矩阵划分为b个带、每个带r行，计算每个带的桶ID，生成<文档唯一标识、签名向量、桶ID> 三元组；

- 所有计算均在本地GPU通过CUDA核函数并行完成，利用FED的通信-计算重叠、共享内存tiling等优化。

1. **步骤3：本地去重与候选集提取**

各机构按FED原版完成**本地语料的全量去重**（GPU矩阵化相似度对比、并查图构建、重复文档删除），消除自身语料的冗余；

同时，按**全局桶ID**对本地去重后的语料做分类，提取每个桶ID下的**文档唯一标识+签名向量**，形成**本地桶候选集**（仅保留本地语料中属于各全局桶的特征），为向主节点提交做准备。

### 阶段2：主节点协同计算（核心隐私保护阶段，主节点统筹完成）

此阶段是该方案的**隐私保护核心**，核心目标是**在不泄露明文特征的前提下，由主节点完成所有跨机构同桶文档的相似度验证**，各机构仅向主节点传输**密文+无语义ID**，不与其他机构直接交互。

1. **步骤4：本地候选集密文化与向主节点提交**

2. **GPU并行密文化**：各机构对本地桶候选集中的**签名向量**做**轻量对称加密**（推荐AES-128/256），并将**文档唯一标识**替换为**无语义随机ID**（如UUID，无法反推文档内容/所属机构）；

3. **向主节点提交密文候选集**：各机构将处理后的<无语义随机ID、密文签名向量、桶ID、机构标识（匿名化）> 四元组，统一传输至联邦主节点，传输过程采用加密协议（TLS 1.3），防止数据截获。

4. **步骤5：主节点密文处理与全局GPU相似度验证**

5. **主节点数据分类**：主节点接收所有机构的密文候选集后，按**全局桶ID**对数据做分类，同一桶ID的所有机构密文候选集归为一组，便于后续相似度对比；

6. **主节点GPU端到端密文处理与相似度对比**：
           

   1. 主节点通过GPU并行解密核函数，对每组同桶密文候选集完成解密（仅解密当前桶的密文，其他桶密文不处理），解密权限受严格管控；

   2. 解密后，主节点**完全复用FED原版的GPU矩阵化相似度对比核函数**，对同一桶内所有机构的文档做**O(B²)全量成对比较**（B为同桶文档总数），计算签名相似度；

   3. 仅保留**相似度≥全局阈值θ（如0.8）的跨机构重复文档对**，其余密文签名/解密后的明文签名立即销毁（不做任何持久化存储），仅留存重复文档的无语义随机ID、机构标识，进一步降低隐私泄露风险；

### 阶段3：结果分发与本地二次去重（主节点分发，各机构执行）

1. **步骤6：主节点结果拆分与分发，各机构完成全局去重**

2. 主节点汇总所有跨机构重复文档对，按**匿名化机构标识**拆分，为每个机构生成“本机构需删除的重复文档无语义随机ID列表”；

3. 主节点将拆分后的列表分别传输至对应机构，传输过程采用加密协议，确保结果不被截获或篡改；

4. 各机构将“重复文档无语义随机ID”映射回“本地文档原始索引”，汇总**本地去重结果+主节点下发的跨机构重复结果**，得到全局去重后的语料索引列表；

5. 各机构根据全局索引列表，在本地删除跨机构重复的文档，完成最终的全局去重；

6. 整个过程中，主节点仅向各机构传输“本机构的重复文档索引”，不传输其他机构的任何特征/语料内容，实现隐私闭环。

## 三、方案的核心隐私保护机制

该方案通过**四层保护机制**实现隐私保障，即使传输数据被截获或主节点出现轻微风险，攻击者也无法反推原始语料/明文特征，具体如下：

### 1. 最小信息传输原则

各机构仅向主节点传输<无语义随机ID、密文签名向量、桶ID、匿名化机构标识>，无任何原始语料、文档语义信息、完整签名矩阵，传输数据为“无意义密文+纯数字/随机标识”，即使被截获也无利用价值。

### 2. 桶级别的隐私隔离

主节点按全局桶ID对密文候选集分类处理，同一桶的文档才会进行跨机构相似度对比，各机构的文档仅与同桶其他机构的文档匹配，无法获取其他桶的任何数据，大幅缩小隐私泄露的范围；且桶数K按FED的K=k√N动态调整（如30B-token语料k=2），单桶文档数少，即使某一桶数据出现风险，影响也极其有限。

### 3. 即时销毁机制

主节点对密文解密后，仅在GPU共享内存中存储明文签名，相似度计算完成后，非重复的明文签名、密文签名立即销毁，不做任何持久化存储，仅保留重复文档的索引信息，避免明文特征在主节点内存中留存导致的泄露风险。

### 4. 轻量对称加密与主节点安全防护

采用的AES-128/256对称加密是工业级隐私保护算法，破解难度极高；且加密密钥由各机构本地独立持有，不向主节点或其他机构传输，即使主节点获取密文，无密钥也无法解密出明文签名向量；同时主节点通过多重硬件与权限防护，规避单点安全风险。

## 四、与FED原版的兼容性与GPU优化

该方案对FED原版的**改造成本低、兼容性高**，仅在**跨机构数据交互层**和**主节点计算层**新增少量逻辑，**底层GPU计算逻辑完全复用**，并通过3个关键优化抵消隐私操作与主节点计算的性能损耗，确保整体效率仅下降10%~15%：

### 1. 完全复用FED的GPU核心计算逻辑

- 各机构本地的shingling、哈希生成、签名矩阵构建、LSH桶划分、矩阵化相似度对比、并查图构建等**GPU密集型操作**，完全复用FED原版的CUDA核函数、共享内存tiling、通信-计算重叠等优化；

- 主节点的全局相似度对比仍采用FED的**O(B²)全量成对比较**，保证去重精度与原版一致，无假阴性增加。

### 2. 隐私操作与主节点计算的GPU并行化

- 各机构的对称加密/解密操作实现为**定制化CUDA核函数**，按FED的线程块/线程划分规则，让每个GPU线程处理一个签名向量的加解密，利用GPU的千级并行性抵消加解密的计算开销，加解密总时间占比＜5%；

- 主节点的密文解密、全局相似度对比，采用GPU集群并行计算，按桶ID拆分任务，每个GPU处理一个或多个桶的计算，利用FED的GPU并行架构，抵消全局计算的性能压力。

### 3. 复用FED的并行I/O与通信优化

- 各机构向主节点传输密文候选集，采用**FED的C桶并行处理规则**，批量传输同桶数据，减少网络I/O次数，适配InfiniBand等高速网络（与FED实验环境的网络配置一致）；

- 将**各机构向主节点的密文传输**，与**本地GPU的相似度计算**做**通信-计算重叠**，在GPU计算本地文档相似度的同时，完成密文数据向主节点的传输，隐藏网络传输延迟。

该方案兼顾隐私性、效率与精度，核心指标如下，对比FED原版无精度损失，性能损耗可控：

| 指标                | 具体数值/特征                                   | 对比FED原版                            |
| ------------------- | ----------------------------------------------- | -------------------------------------- |
| 去重精度（Jaccard） | ≥0.96（与标准MinHash算法对比）                  | 完全一致，无精度损失                   |

| 隐私性              | 中高（工业级加密+桶级隔离+最小传输+主节点防护） | 新增工业级隐私保护机制，远高于原FED    |
| 实现难度            | 中等                                            | 仅新增主节点计算层与交互层，无底层改造 |

| 可扩展性            | 支持动态增删合作机构，主节点可弹性扩容          | 优于FED原版（仅支持单机构扩展）        |

## 六、方案的关键注意事项

1. **全局参数一致性**：所有合作机构必须严格遵守主节点下发的FED超参数（哈希参数p/q、LSH b/r、桶数K、相似度阈值θ），否则会导致跨机构桶ID不匹配，主节点无法完成全局相似度匹配；

2. **密钥管理**：对称加密的密钥由各机构**本地独立管理**，禁止向主节点或其他机构交换密钥，建议采用硬件加密模块（HSM）存储密钥，进一步提升密钥安全性；

3. **无语义ID映射**：各机构需维护<无语义随机ID、本地原始文档索引> 的映射表，且该映射表仅在本地存储，禁止向主节点或其他机构传输，避免ID映射泄露文档信息；

4. **网络传输安全**：各机构与主节点的密文数据交换、结果分发，必须采用**加密传输协议**（如TLS 1.3），防止数据在传输过程中被截获/篡改；

5. **桶数K的动态调整**：桶数K由主节点按FED的K=k√N（k值根据跨机构语料总规模确定）动态调整，避免k值过小导致单桶文档数过多（增加主节点计算开销）或k值过大导致桶数过多（增加传输次数）；

6. **主节点安全防护**：主节点需部署硬件加密模块（HSM）、多权限分级管理、操作审计日志、主备双节点冗余等防护措施，避免单点故障或恶意攻击导致的隐私泄露；

7. **主节点性能扩容**：当跨机构语料规模增长时，需及时对主节点GPU集群进行弹性扩容，避免主节点成为性能瓶颈，影响全局去重效率。

## 七、方案核心价值（与原FED对比）

相较于原FED单机构本地去重方案，本中心化方案的核心优势与延伸价值体现在：

1. **场景延伸**：原FED仅支持单机构本地语料去重，无法解决跨机构语料的跨库冗余问题；本方案实现跨机构中心化协同去重，突破数据孤岛，可支撑多机构大模型语料共建场景，且主节点统筹管理，降低各机构的协同成本。

2. **隐私增强**：原FED无隐私保护机制，仅适用于本地语料处理；本方案通过密文交换、桶级隔离、主节点安全防护等机制，实现跨机构数据交互的隐私闭环，满足跨机构协同的隐私合规要求。

3. **优势保留**：完全复用原FED的GPU加速核心、可复用哈希、矩阵化相似度对比等优化，去重精度与原FED完全一致（Jaccard≥0.96），仅增加10%~15%的性能损耗，兼顾效率与隐私。

4. **工程落地性提升**：中心化架构由主节点统一统筹参数、汇总数据、计算结果、分发任务，无需各机构两两协同，降低了跨机构的协同复杂度，更便于工程化部署与运维。

综上，联邦LSH桶划分+候选集密文交换（中心化版本）是**FED从单机构本地工具向跨机构联邦工具延伸的最优落地方案**，既保留了FED原版的GPU加速核心优势，又通过中心化架构简化跨机构协同流程，同时通过完善的隐私保护机制保障数据安全，在大模型语料共建、跨机构联邦训练等场景中具有极高的落地价值。
